# Django Project Scaffold Reference

*Authoritative preferences document for building a Django bootstrap skill*

---

## Decision Tree: First Question

The first question any scaffold tool should ask is **project type**, because it drives most downstream decisions:

1. **Django full-stack** — Django templates + Tailwind CSS, possibly HTMX. Server-rendered UI. May or may not have a REST API layer.
2. **Wagtail CMS full-stack** — Same as above but built on Wagtail. Rich content editing, media management, page tree.
3. **Django REST API** — DRF backend only. Separate frontend (React, Next.js, mobile, etc.). No Django templates beyond admin.

These types are related but diverge enough to warrant distinct templates. Most structural decisions flow from this choice.

---

## Settled Decisions (Not Up for Debate in Scaffold)

These are confirmed preferences — the scaffold should use them without asking:

| Area | Decision |
| ---- | -------- |
| Deployment target | **Railway** |
| Media storage | **S3 with presigned URLs** (django-storages[s3]) |
| Error monitoring | **Sentry SDK** |
| CI/CD | **GitHub Actions** |
| Pre-commit in CI | **Yes** — same hooks run in CI as locally |
| Frontend styling | **Tailwind CSS** (via npm, not containerized) |
| Python version | **latest stable** (look up at generation time) |
| Django version | **latest LTS** — x.2 series (look up at generation time) |
| Dependency management | **pyproject.toml + uv** |
| Version control | **Git** with `.gitignore` |

## Containerization Philosophy

**Run Django and npm directly on the host. Containerize the services.**

This is the current preferred local dev setup — not a monolithic docker-compose that wraps everything including Django.

**What runs in Docker (always-on services):**

- PostgreSQL (or PostGIS for geodjango projects)
- pgAdmin
- Mailpit (SMTP catching)
- Redis (only if justified — see below)
- Celery worker (only if justified)
- Celery Beat (only if periodic tasks needed)
- Flower (Celery monitoring, only if Celery present)

**What runs directly in terminal:**

- `python manage.py runserver` (or `uvicorn`)
- `npm run dev:css` (Tailwind CSS watch)

**Rationale**: Running Django commands (`migrate`, `createsuperuser`, `shell`, `test`) is far easier without a container layer. `docker exec` adds friction that slows development. npm/Tailwind similarly benefits from direct access.

**Implication for docker-compose.yml**: The `web` service should exist but be optional/commented, or the compose file should only contain supporting services. The compose file should expose database port for direct access from host.

## Settings Architecture

**Always split: `base.py` + `local.py` + `production.py`** — even for API projects. A monolithic settings file creates pain as projects grow.

### base.py — canonical patterns

```python
from pathlib import Path
import environ

BASE_DIR = Path(__file__).resolve().parent.parent.parent

env = environ.Env()
environ.Env.read_env(BASE_DIR / ".env")

SECRET_KEY = env("SECRET_KEY")
DEBUG = env.bool("DEBUG", default=False)

INSTALLED_APPS = [
    # Local apps — first, so their templates take precedence
    "users",
    # Third-party — allauth must come before django.contrib.admin for template resolution
    "allauth",
    "allauth.account",
    # Django
    "django.contrib.admin",
    "django.contrib.auth",
    "django.contrib.contenttypes",
    "django.contrib.sessions",
    "django.contrib.messages",

    "django.contrib.sites",
]

# Always set explicitly
DEFAULT_AUTO_FIELD = "django.db.models.BigAutoField"
AUTH_USER_MODEL = "users.User"  # Always a custom user model — flat app, no apps. prefix

# Database — always via dj-database-url
import dj_database_url

    "default": dj_database_url.config(
        default=env("DATABASE_URL", default="sqlite:///db.sqlite3"),

# Static & Media

MEDIA_URL = "/media/"

# Internationalization

# projects don't use.
```

### production.py — the authoritative template

This is where most inconsistency lived. The canonical version:

```python
import os
from urllib.parse import urlparse
from django.core.exceptions import ImproperlyConfigured
from .base import *

# ─── Validation ───────────────────────────────────────────────────────────────
# Fail loudly at startup if critical env vars are missing.
_required_vars = ["SECRET_KEY", "DATABASE_URL"]
for _var in _required_vars:
    if not os.environ.get(_var):
        raise ImproperlyConfigured(f"Environment variable {_var!r} is required in production.")

SECRET_KEY = os.environ["SECRET_KEY"]
DEBUG = False

# ─── Hosts ────────────────────────────────────────────────────────────────────
# Build ALLOWED_HOSTS from env var (comma-separated) + Railway-injected domains.
# Never use wildcards. Never hardcode domains in code.
ALLOWED_HOSTS = [h.strip() for h in os.environ.get("ALLOWED_HOSTS", "").split(",") if h.strip()]

for _railway_var in ("RAILWAY_PUBLIC_DOMAIN", "RAILWAY_PRIVATE_DOMAIN"):
    if _host := os.environ.get(_railway_var):
        if _host not in ALLOWED_HOSTS:
            ALLOWED_HOSTS.append(_host)

if not ALLOWED_HOSTS:
    raise ImproperlyConfigured("ALLOWED_HOSTS must not be empty in production.")

# ─── CSRF ─────────────────────────────────────────────────────────────────────
# Build CSRF_TRUSTED_ORIGINS from env var + auto-derived Railway/SITE_URL origin.
CSRF_TRUSTED_ORIGINS = [
    o.strip() for o in os.environ.get("CSRF_TRUSTED_ORIGINS", "").split(",") if o.strip()
]

# Auto-derive from SITE_URL (custom domain) first, then fall back to Railway domain.
if _site_url := os.environ.get("SITE_URL"):
    _parsed = urlparse(_site_url)
    _origin = f"{_parsed.scheme}://{_parsed.netloc}"
    if _parsed.scheme and _parsed.netloc and _origin not in CSRF_TRUSTED_ORIGINS:
        CSRF_TRUSTED_ORIGINS.append(_origin)
elif _railway_domain := os.environ.get("RAILWAY_PUBLIC_DOMAIN"):
    _origin = f"https://{_railway_domain}"
    if _origin not in CSRF_TRUSTED_ORIGINS:
        CSRF_TRUSTED_ORIGINS.append(_origin)

# ─── Database ─────────────────────────────────────────────────────────────────
import dj_database_url
DATABASES = {
    "default": dj_database_url.config(
        default=os.environ["DATABASE_URL"],
        conn_max_age=600,
        conn_health_checks=True,
    )
}

# ─── Cache ────────────────────────────────────────────────────────────────────
# Use Redis if available; fall back to locmem. Don't add Redis unless needed.
if _redis_url := os.environ.get("REDIS_URL"):
    CACHES = {
        "default": {
            "BACKEND": "django.core.cache.backends.redis.RedisCache",
            "LOCATION": _redis_url,
        }
    }
else:
    CACHES = {
        "default": {"BACKEND": "django.core.cache.backends.locmem.LocMemCache"}
    }

# ─── Static files ─────────────────────────────────────────────────────────────
STORAGES = {
    "default": {
        "BACKEND": "storages.backends.s3boto3.S3Boto3Storage",  # Media → S3
    },
    "staticfiles": {
        "BACKEND": "whitenoise.storage.CompressedManifestStaticFilesStorage",
    },
}
MIDDLEWARE.insert(1, "whitenoise.middleware.WhiteNoiseMiddleware")

# S3 for media (presigned URLs)
AWS_STORAGE_BUCKET_NAME = os.environ.get("AWS_STORAGE_BUCKET_NAME")
AWS_S3_REGION_NAME = os.environ.get("AWS_S3_REGION_NAME", "ca-central-1")
AWS_ACCESS_KEY_ID = os.environ.get("AWS_ACCESS_KEY_ID")
AWS_SECRET_ACCESS_KEY = os.environ.get("AWS_SECRET_ACCESS_KEY")
AWS_QUERYSTRING_AUTH = True           # Presigned URLs
AWS_S3_FILE_OVERWRITE = False         # Don't overwrite files with same name
AWS_DEFAULT_ACL = None                # Don't set ACL (bucket handles it)

# ─── Email ────────────────────────────────────────────────────────────────────
# Mailtrap via django-anymail. Falls back to SMTP if token not set.
if _mailtrap_token := os.environ.get("MAILTRAP_API_TOKEN"):
    EMAIL_BACKEND = "anymail.backends.mailtrap.EmailBackend"
    ANYMAIL = {"MAILTRAP_API_TOKEN": _mailtrap_token}
else:
    EMAIL_BACKEND = "django.core.mail.backends.smtp.EmailBackend"
    EMAIL_HOST = os.environ.get("EMAIL_HOST", "live.smtp.mailtrap.io")
    EMAIL_PORT = int(os.environ.get("EMAIL_PORT", "587"))
    EMAIL_USE_TLS = True
    EMAIL_HOST_USER = os.environ.get("EMAIL_HOST_USER", "api")
    EMAIL_HOST_PASSWORD = os.environ.get("EMAIL_HOST_PASSWORD", "")
    EMAIL_TIMEOUT = 60

DEFAULT_FROM_EMAIL = os.environ.get("DEFAULT_FROM_EMAIL", "noreply@example.com")
SERVER_EMAIL = DEFAULT_FROM_EMAIL

# ─── Security ─────────────────────────────────────────────────────────────────
SECURE_PROXY_SSL_HEADER = ("HTTP_X_FORWARDED_PROTO", "https")  # Railway proxy
SECURE_SSL_REDIRECT = True
SECURE_REDIRECT_EXEMPT = [r"^health/$"]  # Railway health check uses HTTP — must exempt

SECURE_HSTS_SECONDS = 31_536_000       # 1 year
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SECURE_HSTS_PRELOAD = True
SECURE_CONTENT_TYPE_NOSNIFF = True
SECURE_REFERRER_POLICY = "strict-origin-when-cross-origin"
X_FRAME_OPTIONS = "DENY"
SESSION_COOKIE_SECURE = True
CSRF_COOKIE_SECURE = True

# ─── Logging ──────────────────────────────────────────────────────────────────
# Console logging for Railway (stdout → Railway log drain).
_LOG_LEVEL = os.environ.get("DJANGO_LOG_LEVEL", "INFO").upper()
LOGGING = {
    "version": 1,
    "disable_existing_loggers": False,
    "formatters": {
        "verbose": {
            "format": "[{levelname}] {asctime} {name} {process:d} {message}",
            "style": "{",
        },
    },
    "handlers": {
        "console": {
            "class": "logging.StreamHandler",
            "stream": "ext://sys.stdout",
            "formatter": "verbose",
        },
    },
    "root": {"handlers": ["console"], "level": _LOG_LEVEL},
    "loggers": {
        "django": {"handlers": ["console"], "level": _LOG_LEVEL, "propagate": False},
        "django.security.DisallowedHost": {"handlers": [], "propagate": False},  # Don't log bot probes
    },
}

# ─── Sentry ───────────────────────────────────────────────────────────────────
if _sentry_dsn := os.environ.get("SENTRY_DSN"):
    import sentry_sdk
    from sentry_sdk.integrations.django import DjangoIntegration
    from django.core.exceptions import DisallowedHost

    def _before_send(event, hint):
        """Drop DisallowedHost noise from bots probing with wildcard hosts."""
        exc_info = hint.get("exc_info")
        if exc_info:
            exc_type, _, _ = exc_info
            if exc_type and issubclass(exc_type, DisallowedHost):
                return None
        return event

    sentry_sdk.init(
        dsn=_sentry_dsn,
        integrations=[DjangoIntegration()],
        traces_sample_rate=float(os.environ.get("SENTRY_TRACES_SAMPLE_RATE", "0.1")),
        send_default_pii=False,
        before_send=_before_send,
        environment=os.environ.get("RAILWAY_ENVIRONMENT", "production"),
    )
```

## The Health Check Endpoint

Railway's health check **uses HTTP, not HTTPS**. Django's `SECURE_SSL_REDIRECT = True` will redirect it, causing a timeout loop. The fix is mandatory and must be in every project.

**In `config/urls.py`:**

```python
from django.http import JsonResponse

def health_check(request):
    return JsonResponse({"status": "ok"}, status=200)

urlpatterns = [
    path("health/", health_check, name="health-check"),
    # ... rest of urls
]
```

**In `config/settings/production.py`:**

```python
SECURE_REDIRECT_EXEMPT = [r"^health/$"]
```

**In `railway.json`** (under the `deploy` key):

```json
"healthcheckPath": "/health/"
```

**Pattern notes:**

- Use `^health/$` (with trailing slash anchor) — more specific than `^health/`
- Return `{"status": "ok"}` with status 200
- Do not check DB or external services in the health endpoint — keep it simple; Railway just needs a 200
- The URL must be registered *before* any URL patterns that might catch-all

## Deployment: railpack.json

Railway deprecated nixpacks in favor of Railpack. All new projects use Railpack. **Do not generate a `nixpacks.toml`.**

`railpack.json` is intentionally minimal — it only declares runtime versions. Everything else goes in `railway.json`.

```json
{
  "packages": {
    "node": "<latest LTS>",
    "python": "<latest stable>"
  }
}
```

Look up the current Node.js LTS and Python stable versions at generation time — do not hardcode stale version strings.

GeoDjango on Railway (PostGIS) requires additional apt packages in `railpack.json`, but this configuration has not been tested with Railpack. Do not scaffold it until a working deployment has been confirmed.

### Multilingual projects (Q7 has more than one language)

Two extra requirements when `compilemessages` is needed on Railway:

**1. `railpack.json` — add `gettext` apt package** (required for `compilemessages` to work):

```json
{
  "packages": {
    "node": "<latest LTS>",
    "python": "<latest stable>"
  },
  "buildAptPackages": ["gettext"],
  "deploy": {
    "aptPackages": ["gettext"]
  }
}
```

**2. `railway.json` — add `compilemessages` to `startCommand`**, before gunicorn, with `|| true` so a missing locale directory does not abort the deploy:

```json
"startCommand": "DJANGO_SETTINGS_MODULE=config.settings.production python manage.py migrate --noinput && python manage.py compilemessages --locale en --locale {lang2} || true && gunicorn config.wsgi:application --bind 0.0.0.0:${PORT} --workers 3 --access-logfile - --error-logfile -"
```

Replace `{lang2}` with each additional locale code (e.g. `--locale fr`). Add one `--locale` flag per language.

The `|| true` is important — if `locale/` doesn't exist yet or `.po` files haven't been created, `compilemessages` exits non-zero and would kill the deploy without it.

## Deployment: railway.json

This is the primary deployment configuration file. The canonical pattern:

```json
{
  "$schema": "https://railway.com/railway.schema.json",
  "build": {
    "builder": "RAILPACK",
    "buildCommand": "npm install && npm run build:css && python manage.py collectstatic --noinput"
  },
  "deploy": {
    "runtime": "V2",
    "numReplicas": 1,
    "startCommand": "DJANGO_SETTINGS_MODULE=config.settings.production python manage.py migrate --noinput && gunicorn config.wsgi:application --bind 0.0.0.0:${PORT} --workers 3 --access-logfile - --error-logfile -",
    "healthcheckPath": "/health/",
    "sleepApplication": false,
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 10
  }
}
```

For API-only projects (no npm/Tailwind), simplify the buildCommand:

```json
"buildCommand": "python manage.py collectstatic --noinput"
```

**Gunicorn flags:**

- `--bind 0.0.0.0:${PORT}` — required; Railway injects `$PORT`
- `--workers 3` — standard for Railway starter/hobby plans
- `--access-logfile -` and `--error-logfile -` — log to stdout; Railway reads stdout

**`DJANGO_SETTINGS_MODULE` in `startCommand`**: set it explicitly here even if it is also set as a Railway env var — belt and suspenders for a critical setting.

**Do not add** `--timeout 120` unless you have evidence of timeouts. Default gunicorn timeout (30s) is fine for most apps.

**Do not add** `multiRegionConfig` to the scaffold — it is infrastructure-specific and set per-project after deployment.

```yaml
    rev: 1.23.0

  - repo: https://github.com/Riverside-Healthcare/djLint

    hooks:
      - id: djlint-reformat-django       # Format Django HTML templates
```

- `mypy` is included by default — it catches real bugs and pays for the setup cost quickly.
- `ripsecrets` (prevent committing secrets) is a good addition for projects with many env vars.
- For projects using sqlfluff: add it as an optional hook (especially projects with complex raw SQL or Wagtail migrations).

## Authentication

Never use `AUTH_USER_MODEL = "auth.User"` — changing it after the first migration is painful.

### users/models.py

Email-based auth from the start: `username` removed, `email` is the login field. No forced first/last name.

```python
from django.contrib.auth.models import AbstractUser
from django.contrib.auth.models import UserManager as DjangoUserManager
from django.db import models


class UserManager(DjangoUserManager):
    def create_user(self, email, password=None, **extra_fields):
        if not email:
            raise ValueError("The Email field must be set")
        email = self.normalize_email(email)
        user = self.model(email=email, **extra_fields)
        user.set_password(password)
        user.save(using=self._db)
        return user

    def create_superuser(self, email, password=None, **extra_fields):
        extra_fields.setdefault("is_staff", True)
        extra_fields.setdefault("is_superuser", True)
        extra_fields.setdefault("is_active", True)
        if extra_fields.get("is_staff") is not True:
            raise ValueError("Superuser must have is_staff=True.")
        if extra_fields.get("is_superuser") is not True:
            raise ValueError("Superuser must have is_superuser=True.")
        return self.create_user(email, password, **extra_fields)


class User(AbstractUser):
    username = None
    email = models.EmailField("email address", unique=True)

    USERNAME_FIELD = "email"
    REQUIRED_FIELDS = []

    objects = UserManager()
```

### users/tests/test_users.py

```python
from django.contrib.auth import get_user_model
from django.test import TestCase

User = get_user_model()


class TestUserManager(TestCase):
    def test_create_user_requires_email(self):
        with self.assertRaisesMessage(ValueError, "The Email field must be set"):
            User.objects.create_user(email=None, password="secret")

    def test_create_user_normalizes_email_and_sets_password(self):
        user = User.objects.create_user(email="TEST@EXAMPLE.COM", password="secret")
        self.assertEqual(user.email, "TEST@example.com")
        self.assertTrue(user.check_password("secret"))
        self.assertFalse(user.is_staff)
        self.assertFalse(user.is_superuser)
        self.assertTrue(user.is_active)

    def test_create_superuser_enforces_required_flags(self):
        with self.assertRaisesMessage(ValueError, "Superuser must have is_staff=True."):
            User.objects.create_superuser(email="a@example.com", password="secret", is_staff=False)
        with self.assertRaisesMessage(ValueError, "Superuser must have is_superuser=True."):
            User.objects.create_superuser(email="b@example.com", password="secret", is_superuser=False)

    def test_create_superuser_sets_defaults(self):
        admin = User.objects.create_superuser(email="admin@example.com", password="secret")
        self.assertTrue(admin.is_staff)
        self.assertTrue(admin.is_superuser)
        self.assertTrue(admin.is_active)
```

**allauth decision:**

| Scenario | Package |
| -------- | ------- |
| Full-stack, social auth possible later | `django-allauth` |
| Full-stack, Google OAuth needed now | `django-allauth[socialaccount]` |
| REST API | `djangorestframework-simplejwt` |
| Built-in only | no extra package |

## Redis & Celery Decision Framework

### Redis

**Don't add Redis by default.** Add it only when one of the following is true:

- Celery is being used (needs a broker)
- Session data is large or multi-server (need distributed sessions)
- Rate limiting is needed (cache-based)
- Cache invalidation strategy has been designed

**If Redis is added**, use the graceful fallback pattern in production.py (shown above). In local dev, no need for Redis unless testing Redis-specific behavior.

### Celery

**Don't add Celery by default.** Add it only when:

- Background jobs are genuinely long-running (>5s) — e.g., PDF generation, AI calls, image processing
- Tasks need to be retried on failure
- Tasks need to run on a schedule (`django-celery-beat`)
- Volume of emails is high enough that synchronous sending is a UX problem

**What doesn't need Celery:**

- Sending a single transactional email (password reset, welcome) — synchronous is fine
- Simple data processing that runs in under 1s
- File uploads to S3 (django-storages handles this synchronously and it's fast enough)

**When Celery is added**, always use `config/celery.py` (not inline in settings):

```python
# config/celery.py
import os
from celery import Celery

os.environ.setdefault("DJANGO_SETTINGS_MODULE", "config.settings.local")

app = Celery("project_name")
app.config_from_object("django.conf:settings", namespace="CELERY")
app.autodiscover_tasks()
```

And in `config/__init__.py`:

```python
from .celery import app as celery_app
__all__ = ["celery_app"]
```

## Caching Strategy

**Local dev**: `locmem` cache always. Never spin up Redis just for local dev unless testing cache behavior specifically.

**Production**: If Redis is available (because Celery is present), use it for caching too. Otherwise, locmem is fine for low-traffic projects — it's per-process but acceptable until traffic warrants Redis.

**Things worth caching** (when you do add Redis):

- Rendered Wagtail page fragments
- API responses for expensive aggregation queries
- Session data (if using database sessions, Redis sessions are faster at scale)

**Things not worth caching at this traffic level:**

- Individual model queries
- Per-user data (tricky invalidation, low benefit)

## HTMX and Alpine.js

Both are served from `static/js/` as downloaded minified files. No CDN. No pip package.

Download once and commit:

```text
static/js/htmx.min.js
static/js/alpine.min.js
```

Add to `base.html` (inside `<head>`, using Django's `{% static %}` tag):

```html
{% load static %}
<!-- HTMX -->
<script src="{% static 'js/htmx.min.js' %}" defer></script>
<!-- Alpine.js — defer required; must load before Alpine initializes -->
<script src="{% static 'js/alpine.min.js' %}" defer></script>
```

**`django-htmx` is optional.** It adds Django-specific middleware and request helpers but is not required for HTMX to work. Include it only if you are using its specific features (e.g., `request.htmx`, `HtmxResponseMixin`).

When Alpine and HTMX are used together, no special integration is needed for most cases. If Alpine components live inside HTMX-swapped fragments, use `x-data` on a stable ancestor outside the swap target.

No `INSTALLED_APPS` entry for either library.

## tailwind.config.js

**Tailwind v4 does not require a `tailwind.config.js`** — do not generate one unless the project explicitly uses v3 or needs a JS config for advanced plugin usage.

In v4, configuration lives in the CSS file via `@theme` blocks. If a project does need a JS config (e.g., for `@tailwindcss/typography` plugin or complex theme customization), it is loaded from the CSS file with `@config "./tailwind.config.js"`.

For v3 projects only, the canonical config:

```js
/** @type {import('tailwindcss').Config} */
module.exports = {
  content: [
    "./templates/**/*.html",
    "./**/*.html",
    "./**/*.py",
    "./static/**/*.js",
  ],
  theme: {
    extend: {},
  },
  plugins: [],
}
```

## package.json

The Tailwind **input** file lives in `assets/css/`, not `static/`. This avoids a `collectstatic` conflict where both the source and compiled CSS would be collected, causing errors with `CompressedManifestStaticFilesStorage`.

`assets/` is a source-only directory — never added to `STATICFILES_DIRS`. The compiled output in `static/css/` is what Django serves.

```json
{
  "name": "{project-slug}",
  "private": true,
  "scripts": {
    "dev:css": "tailwindcss -i ./assets/css/input.css -o ./static/css/tailwind.css --watch",
    "build:css": "tailwindcss -i ./assets/css/input.css -o ./static/css/tailwind.css --minify"
  },
  "devDependencies": {
    "@tailwindcss/cli": "^4.x.x",
    "tailwindcss": "^4.x.x"
  }
}
```

**Tailwind v4** is the current default. In v4, `@tailwindcss/cli` is a separate package required for the CLI. No `tailwind.config.js` is needed for basic projects — configuration lives in the CSS file via `@theme` blocks.

## Known Deployment Bugs Fixed by This Standard

The following are common deployment bugs addressed by this document:

| Bug | Fix |
| --- | --- |
| Missing `--bind 0.0.0.0:$PORT` in gunicorn | Include in startCommand; Railway may handle it automatically with Railpack but be explicit |
| No health check endpoint | Required: `/health/` URL + view |
| Missing `SECURE_REDIRECT_EXEMPT` | Always add `[r"^health/$"]` |
| Wildcard `.railway.app` in ALLOWED_HOSTS | Build from env var + Railway vars |
| No `CSRF_TRUSTED_ORIGINS` auto-derivation | Use SITE_URL + RAILWAY_PUBLIC_DOMAIN |
| No production logging config | Always define LOGGING in production.py |
| No email backend in production | Always define with Mailtrap fallback |
| No Sentry | Always include, gated on SENTRY_DSN |
| `traces_sample_rate=1.0` (100% trace) | Default 0.1; configurable via env |
| Missing `conn_health_checks=True` | Always include in database config |
| Inconsistent `SECURE_HSTS_*` | Standardized in this document |
| JSON parsing left in build command | Build command should be clean |
| `gunicorn` without timeout | Always `--timeout 120` |
| No production.py validation | Always validate required env vars |
